# 任务归档仓库

这是一个轻量的任务/需求归档仓库，用来保存不同需求的原始信息、独立代码、说明文档和相关附件。

## 设计原则

- 每个任务必须独立成包
- 每个任务目录都可以单独打包或直接发给外部
- 不依赖仓库内其他任务目录
- 不设置跨任务共享代码目录

也就是说，某个任务如果需要脚本、配置、说明文件、截图或示例数据，都应当放在该任务自己的目录里。

## 目录说明

- `entries/`：实际归档内容目录，每个需求或任务一个子目录
- `templates/`：可复用的归档模板目录

## 单个归档条目建议结构

每个归档条目建议使用如下结构：

```text
entries/某个任务名/
├── request/     # 原始需求、沟通记录、背景说明
├── code/        # 该任务独立代码、脚本、配置
├── docs/        # 方案说明、使用说明、交付说明
└── assets/      # 截图、附件、样例文件
```

## 当前已创建内容

- `entries/example-entry/`：示例归档条目
- `templates/entry-template/`：可复制的新条目模板

## 推荐使用方式

1. 收到新需求时，在 `entries/` 下复制一个新目录
2. 将原始说明放入 `request/`
3. 将该任务所需的全部代码和配置放入 `code/`
4. 将结果说明、交付说明、使用方法放入 `docs/`
5. 将截图、附件、样例文件放入 `assets/`

## 归档规则

- 不要让一个任务引用另一个任务目录中的文件
- 不要依赖仓库根目录下的公共脚本或公共组件
- 如有通用代码需要复用，直接复制到当前任务目录并单独维护
- 对外发送时，直接发送对应任务子目录即可

## 命名建议

建议按下面方式命名归档目录，方便后续查找：

- `2026-03-26-任务名称`
- `项目名-需求名称`
- `客户名-问题名称`

如果后续归档量变大，可以再按年份或项目继续分层，但仍建议保持每个任务目录自包含。
